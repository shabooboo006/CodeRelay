# Phase 02.1: App Icon & Multilingual Support - Context

**Gathered:** 2026-04-02
**Status:** Ready for planning

<domain>
## Phase Boundary

Deliver release-identity polish for the existing CodeRelay app by adding a real macOS app icon and localizing the current app shell plus managed-account flows into English and Simplified Chinese. This phase does not add warning logic, switching automation, menu bar UX, or a user-selectable language picker.

</domain>

<decisions>
## Implementation Decisions

### Release Identity
- **D-01:** Ship a real CodeRelay app icon in the packaged `.app` and release artifacts instead of relying on the default executable icon.
- **D-02:** Keep the icon pipeline scriptable inside the current SwiftPM + shell packaging flow; do not introduce an Xcode project requirement just to generate or copy icon assets.
- **D-03:** Scope icon work to the primary app icon only. Menu bar glyphs, alternate icons, and broader branding refresh stay out of scope.

### Localization Architecture
- **D-04:** English remains the base language, and Simplified Chinese (`zh-Hans`) is the first additional locale.
- **D-05:** Follow the system locale automatically; do not add an in-app language switch in this phase.
- **D-06:** Move user-facing copy out of inline hard-coded SwiftUI strings into shared localized app resources that remain compatible with `swift build`, `swift run`, and the existing release packaging scripts.

### Translation Scope
- **D-07:** Localize the current app shell and managed-account experience end-to-end: window title context, section headings, buttons, empty states, monitoring labels, readiness/status copy, and user-visible action/result messages.
- **D-08:** Keep the product name `CodeRelay`, account emails, percentages, timestamps, and other data values untranslated; localize only the surrounding labels and messages.

### Verification Expectations
- **D-09:** Replace brittle source-level assertions that depend on English string literals with localization-aware coverage so English and Simplified Chinese both remain protected.
- **D-10:** Release verification for this phase must include a packaged `CodeRelay.app` check that confirms the icon is present and that the app launches with localized primary copy under both supported locales.

### the agent's Discretion
- Exact icon art direction and source-file format, as long as the pipeline stays reproducible in-repo.
- String key naming, helper organization, and lightweight localization wrappers.
- Locale-sensitive date/time formatting details, provided the result follows system conventions.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `scripts/build_macos_release.sh`: already assembles `CodeRelay.app`, so icon/resource copying should extend this path instead of inventing a second packaging flow.
- `Packaging/macOS/Info.plist`: existing template already owns bundle metadata, which makes it the right place to wire icon and localization defaults.
- `README.md` + `README.zh-CN.md`: provide an existing bilingual terminology source that can reduce inconsistent translations.

### Established Patterns
- The app is a SwiftPM executable target with no `.xcassets`, `.lproj`, or localization catalog checked in yet.
- `AccountsView` currently hard-codes all visible copy directly in SwiftUI literals, so localization will require a systematic extraction pass rather than spot fixes.
- `CodeRelayAppTests` currently protect UI copy through raw source-string checks, which will break once strings move into resources unless the tests are updated.

### Integration Points
- App-localized resources need to live in or alongside `Sources/CodeRelayApp` so SwiftPM can bundle them for local runs and packaged builds.
- The packaging script and `Info.plist` must agree on icon resource names and bundle placement for `CodeRelay.app`.
- Locale-aware formatting should stay close to the existing SwiftUI view helpers so monitoring/status copy does not fork into ad hoc formatting paths.

</code_context>
